# HAIM-MAS 调度建议 MVP 设计文档

**项目:** 高海拔智慧矿山多智能体协作系统 (HAIM-MAS)  
**阶段:** MVP — 智能调度建议  
**日期:** 2026-05-14  
**版本:** v1.1（基于 plan-ceo-review 建议更新）  
**模式:** Intrapreneurship (内部项目快速交付)  
**基于:** office-hours 设计思维流程 + plan-ceo-review

---

## 1. 项目背景与问题定义

### 核心痛点

- 矿山年产值 **2.6亿人民币**，完全按产量计费
- 设备综合效率 (OEE) 仅 **50-60%**，40%+ 产能未被利用（基线：取近3个月OEE均值，目标提升+15%）
- 每月安全停工约 **1天**，直接损失约 **852万/年**
- 调度全靠对讲机 + 现场拍脑袋，无任何电子记录
- 驾驶员可作弊（空踩油门假装工作），信任建立困难
- 项目经理每周被问责，设备/生产/安全三个经理都在忙救火，提不出方向

> **注：** 文档中"50-60%"为范围。基线将于MVP启动前通过信息化负责人获取近3个月实际OEE数据，锁定为单一数字。所有成功指标均基于锁定后的基线计算。

### 根因分析

当前矿山是**应急驱动型系统**：
- 挖掘机随时被抽走处理突发状况（堵破碎口、修伞岩、清边坡等）
- 卡车链因挖掘机临时调走而全面乱套
- 调度员靠对讲机协调，人脑无法全局优化
- 没有调度记录 → 没有训练数据 → 算法无法学习

> **注意：** 调度员的"拍脑袋"经验包含大量隐性知识——哪台铲机效率高、哪个司机靠谱、哪个路段何时易堵。MVP设计不应假设"系统比经验强"，而应将系统定位为**经验的可视化放大镜**：让调度员的直觉被数据验证，让好决策被记录，让坏决策被追溯。第一阶段需对现有调度员决策模式进行观察和访谈，捕获并编码这些经验知识。

### 机会窗口

- GPS + CAN 数据已实时从主机厂平台获取，精度足够（米级GPS，发动机转速/油耗/电耗/状态）
- 现有系统已实现设备可视化 + OEE报表
- 总经理最急的是"OEE从50%往75%走"

## 2. MVP 定义

### 目标

用现有 GPS + CAN 数据，在**数据充分的区域**给出调度建议（human-in-the-loop），不夺权，建立信任，目标是 **OEE 从 50% 提升至 65%+**。

### MVP 范围（3个月，5人以内）

**做：**
- 智能调度建议（系统给最优路径，人决定是否采纳）
- 多源校验（GPS轨迹 + CAN工作状态 + 产量计数，防欺诈）
- 实时设备状态监控（升级现有监控，加入工作状态识别）
- 日调度效果报表（证明系统有贡献）

**不做：**
- 自动驾驶决策（不夺权，人工确认）
- 全局路径优化算法（先做单体最优）
- 预测性维护（后续迭代）

### 成功指标

> **基线锁定：** OEE基线值须在MVP启动前通过信息化负责人获取近3个月实际OEE均值，锁定为单一数字后再对外承诺。

| 指标 | 基线 | MVP目标 | 备注 |
|------|------|---------|------|
| OEE | 待锁定 | +15%（绝对值） | 如基线50%，目标65% |
| 年度增产（人民币） | — | OEE提升对应产值增长 | 2.6亿 × OEE提升比例，是总经理向董事会汇报的核心数字 |
| 卡车有效作业时间占比 | 未知（需基线测量） | +10%（绝对值） | |
| 调度建议采纳率 | N/A（新系统） | >50% | 第二阶段末考核 |
| 因调度导致的停等时间 | 未知 | -15% | |
| 设备异常检出率 | 0 | >60% | 第一阶段末考核，以"设备健康监测"而非"抓作弊"为叙事 |

**撤退条件：**
- 第一阶段末：调度员周活跃<50% → 信任建立失败，需重新设计交互
- 第二阶段末：调度建议采纳率<30% → 建议质量不足，需重新训练算法
- 第三阶段末：OEE提升<5% → MVP方向存疑，需全面复盘

## 3. 数据架构

### 现有数据源

```
┌─────────────────────────────────────────────────────────────┐
│                    主机厂数据平台                            │
│         (GPS实时 + CAN总线数据，含发动机转速/油耗/电耗/状态)  │
└──────────────────────┬──────────────────────────────────────┘
                       │ 实时推送（秒级或分钟级）
                       ▼
┌─────────────────────────────────────────────────────────────┐
│                   本地 PostgreSQL 数据库                     │
│                                                             │
│  表: vehicle_gps                                            │
│    - vehicle_id, timestamp, lat, lon, speed, heading        │
│                                                             │
│  表: vehicle_can                                            │
│    - vehicle_id, timestamp, engine_rpm, fuel_rate,          │
│      battery_soc, work_status (停机/怠速/工作)              │
│                                                             │
│  表: production_record                                      │
│    - vehicle_id, timestamp, loaded, haul_distance           │
│    (人工统计，每日汇总，当前无实时产量数据)                   │
└─────────────────────────────────────────────────────────────┘
```

### 数据质量评估

| 数据源 | 精度 | 频率 | 可用性 | 备注 |
|--------|------|------|--------|------|
| GPS | 米级误差 | 实时 | ✅ 直接获取 | 需确认采样间隔 |
| CAN 发动机转速 | 精确 | 实时 | ✅ 直接获取 | 可区分挖掘/满载/空驶 |
| CAN 油耗/电耗 | 精确 | 实时 | ✅ 直接获取 | 用于欺诈检测 |
| CAN 工作状态 | 停机/怠速/工作 | 实时 | ✅ 已有字段 | 需确认定义 |
| 产量数据 | 人工统计 | 每日汇总 | ⚠️ 需升级 | 建议加装称重传感器 |

### 关键约束

- **欺诈风险**：驾驶员可空踩油门假装工作，发动机转速无法完全防作弊
- **解决方向**：多源校验 = GPS轨迹合理性 + CAN工作状态 + 产量实际增长

## 4. 设备健康监测机制

### 设计原则

系统不直接判定"作弊"，而是**标记异常 + 人工事后复核**：让调度员知道哪些设备可能有问题，最终由人判断。

**叙事调整：** 欺诈检测不面向管理层开放数据，避免调度员将系统视为"监控工具"而抵触系统。系统对外统一使用"设备健康监测"叙事——发现异常是为了提醒设备状态，调度员是设备的守护者而非被监控者。

### 异常检测规则

```python
# 设备健康检测规则（伪代码）

def detect_anomaly(vehicle_id, time_window):
    gps = query_gps(vehicle_id, time_window)
    can = query_can(vehicle_id, time_window)
    production = query_production(vehicle_id, time_window)
    
    # 规则1: GPS显示移动，但CAN显示停机 → 传感器可能异常
    if gps.speed > 5 and can.work_status == "停机":
        flag("GPS-CAN状态不一致，请检查传感器")
    
    # 规则2: CAN显示工作，但GPS显示车辆未移动 → 可能是空踩或传感器异常
    if can.engine_rpm > 1500 and gps.speed == 0 and not is_in_mine_area(gps):
        flag("设备状态与位置不匹配，建议现场确认")
    
    # 规则3: 产量没有增长，但CAN显示满载工作状态持续超阈值 → 可能有计量异常
    if can.状态 == "满载" and production.delta < expected_threshold:
        flag("作业状态与产量不匹配，请确认设备计量")
    
    # 规则4: 发动机高转速 + 低油耗（空踩特征）
    if can.engine_rpm > 2000 and can.fuel_rate > idle_threshold and gps.speed < 2:
        flag("发动机空转时间过长，建议检查作业状态")
    
    return anomalies
```

### 多源校验矩阵

| 异常类型 | GPS | CAN | 产量 | 应对 |
|----------|-----|-----|------|------|
| 空踩油门（疑似） | 速度≈0 | 高转速+高油耗 | 无增长 | 标记 → 调度员以"设备状态确认"名义现场核查 |
| GPS漂移（矿外飘移） | 矿区外 | 工作中 | 有增长 | GPS修正算法 |
| 传感器故障 | 多个指标异常 | — | — | 设备自检报警 |
| 故意绕路 | 轨迹异常 | 油耗偏高 | 正常 | 标记 → 以"油耗异常"名义复核轨迹 |

### 处理流程

```
异常检测 → 系统标记 → 调度员收到提示 → 调度员判断 → 记录结果 → 优化规则
```

所有复核记录存入数据库，用于后续算法迭代。

### MVP 涉及的核心智能体

```
┌─────────────────────────────────────────────────────────────────────┐
│                         HAIM-MAS 系统                               │
│                                                                     │
│  ┌──────────────────┐   ┌──────────────────┐   ┌──────────────────┐ │
│  │   爆堆数字孪生    │   │  交通与环境治理  │   │   设备状态与供应链 │ │
│  │   智能体         │   │   智能体         │   │    智能运维       │ │
│  │  (Blast Pile DT) │   │  (Traffic&Env)   │   │  (Equipment Maint)│ │
│  └────────┬─────────┘   └────────┬─────────┘   └────────┬─────────┘ │
│           │                      │                      │           │
│           └──────────────────────┼──────────────────────┘           │
│                                  │                                  │
│                         ┌────────▼────────┐                         │
│                         │   调度建议      │                         │
│                         │   智能体        │  ← MVP 核心             │
│                         │(Scheduling Adv.)│                         │
│                         └────────┬────────┘                         │
│                                  │                                  │
│                         ┌────────▼────────┐                         │
│                         │   安全与应急   │                         │
│                         │   守卫智能体   │                         │
│                         └────────┬────────┘                         │
└─────────────────────────────────────────────────────────────────────┘
```

### 调度建议智能体工作流程

```
1. 环境感知
   ├── GPS数据 → 车辆实时位置 + 速度
   ├── CAN数据 → 发动机状态 + 工作状态
   ├── 爆堆数字孪生 → 当前铲装点可用性 + 爆堆量
   └── 交通与环境 → 当前路况 + 封路信息

2. 调度计算
   ├── 任务队列：待装载车辆优先级
   ├── 路径优化：最短等待路径
   ├── 冲突检测：某台挖掘机是否被其他任务占用
   └── 建议生成：最优车辆 → 最优铲装点

3. 建议输出
   ├── 调度建议 → 对讲机/移动终端（调度员）
   ├── 异常标记 → 欺诈可疑车辆（调度员复核）
   └── 状态更新 → 爆堆数字孪生（实时同步）

4. 执行反馈
   ├── 调度员确认/否决 → 记录决策
   └── 实际执行结果 → 反馈给系统 → 算法优化
```

### 与其他智能体的交互

**与爆堆数字孪生智能体：**
- 输入：铲装点当前位置、爆堆剩余量、设备效率预测
- 输出：该区域可接纳的车辆数量建议

**与交通与环境治理智能体：**
- 输入：实时路况、封路信息、粉尘浓度
- 输出：建议绕行路径、敏感区域限速

**与设备状态与供应链智能运维：**
- 输入：设备健康状态、保养计划
- 输出：预警车辆（优先派轻活，避免路上抛锚）

**与安全与应急守卫智能体：**
- 输入：气象、地质、安全规章
- 输出：调度限制（如雷雨天禁止某区域作业）

## 6. MVP 三阶段里程碑

### 第0阶段：数据接入验证（1周，先于所有阶段）

> 所有后续工作的基础。数据接不上，后续阶段全部延期。

**目标：** 确认数据能从主机厂平台稳定拉到本地数据库。

**交付物：**
- 数据接口对接测试报告（GPS采样间隔、CAN字段定义）
- 数据质量评估报告（缺失率、延迟、异常值）
- OEE基线数据（近3个月均值，锁定为单一数字）

**成功标准：**
- GPS数据延迟<1分钟，采样间隔稳定
- CAN数据字段与文档一致，能区分停机/怠速/工作
- OEE基线锁定

**撤退条件：**
- 数据接口无法对接 → 暂停MVP，与主机厂协商数据开放方案

### 第一阶段：数据可视化 + 设备健康监测（4-6周）

> 含第0阶段1周，总计5-7周。

**目标：** 让调度员看到实时数据和异常提示，开始建立信任。

**交付物：**
- 实时车辆状态监控大屏（GPS + CAN 工作状态）
- 设备健康监测系统（异常标记 V1，以"设备状态确认"名义推送）
- 调度效果日志（记录每次建议和采纳结果）
- 调度员决策模式观察报告（捕获老调度员经验知识）

**成功标准：**
- 调度员周活跃 > 80%（周均使用超4天）
- 设备异常检出率 > 60%（人工复核确认）
- 无重大漏报（实际异常未被标记）
- **撤退线：调度员周活跃<50% → 信任建立失败，需重新设计交互**

### 第二阶段：智能调度建议（6-8周）

**目标：** 系统能给出优于经验的调度建议，被采纳率 > 50%。

**交付物：**
- 调度建议引擎（单体最优路径）
- 历史调度效果分析报表
- 调度员反馈闭环系统
- 调度员激励机制设计（采纳建议=轻松/好绩效，否决建议=记录理由）

**成功标准：**
- 调度建议采纳率 > 50%
- 因调度导致的停等时间 -10%
- 无因建议错误导致的安全事故
- **撤退线：调度建议采纳率<30% → 建议质量不足，需重新训练算法**

### 第三阶段：OEE 验证（4-6周）

**目标：** 证明系统对 OEE 的贡献，准备扩大范围。

**交付物：**
- OEE 日报/周报（系统自动生成）
- 设备健康监测效果评估报告
- 年度增产计算报告（OEE提升对应的产值增长，人民币）
- MVP 总结 + 下一阶段建议

**成功标准：**
- OEE 提升 +15%（绝对值）
- 设备异常事件减少 > 50%（对比基线）
- 总经理签字确认"继续投入"
- **撤退线：OEE 提升<5% → MVP方向存疑，需全面复盘**

## 7. 技术架构

### 推荐技术栈

| 层级 | 技术选型 | 理由 |
|------|----------|------|
| 数据接入 | MQTT Broker → PostgreSQL | 现有数据库，已支持 |
| 实时处理 | Python后台进程 + Redis队列 | 轻量，5人团队可hold住，Kafka/Flink留作未来扩展 |
| 规则管理 | Python函数 + YAML配置 | 开发快，调试直观，无需学习规则引擎语法 |
| API 服务 | FastAPI (Python) | 快速迭代，文档友好 |
| 前端大屏 | React + Ant Design Pro | 组件丰富，适合监控 |
| 存储 | PostgreSQL + TimescaleDB | 时序数据优化 |
| 缓存 | Redis | 实时状态缓存，规则引擎状态存储 |
| 部署 | Docker Compose (MVP) | 简单，未来可迁移 K8s |

> **扩展性说明：** MVP阶段用Redis队列替代Kafka Streams是工程可行性取舍。若数据量超过10000条/秒，或需要复杂的流处理算子（如时间窗口聚合、状态再处理），再引入Kafka/Flink。

### 关键 API 设计

```
POST /api/v1/scheduling/suggest
  → 输入：vehicle_ids[], task_type, urgency
  → 输出：[{vehicle_id, recommended_zone, confidence, reasons}]

POST /api/v1/scheduling/feedback
  → 输入：suggestion_id, accepted, actual_outcome, notes
  → 输出：{status: "recorded"}

GET /api/v1/anomaly/list
  → 输入：time_range, vehicle_id (optional)
  → 输出：[{anomaly_id, vehicle_id, type, severity, timestamp}]

POST /api/v1/anomaly/review
  → 输入：anomaly_id, is_fraud, notes
  → 输出：{status: "updated", feedback_to_model}
```

## 8. 调度员激励机制设计

> 系统要让调度员用，先要回答：调度员用这个系统，对他有什么好处？

### 调度员采纳系统建议的动机

| 动机类型 | 设计方式 |
|----------|----------|
| 轻松 | 系统给出建议，调度员一键确认，比自己拍脑袋省力 |
| 好绩效 | 采纳建议后的OEE提升，计入调度员KPI（需与管理层确认） |
| 被尊重 | 调度员的否决理由被系统记录，说明系统不是来替代他的 |
| 少背锅 | 有了决策记录，结果可追溯，调度员不再是唯一责任人 |

### 责任边界定义

> 调度决策涉及安全，责任归属必须清晰。

- **系统**：给出建议，记录决策过程和结果
- **调度员**：最终决策，对决策结果负责
- **出了事故**：根据调度日志追溯，系统建议不作为免责依据，但记录可证明调度员已尽合理注意义务

> 此定义需与法务/管理层确认，作为系统上线前的必要条件。

### 主要风险

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 调度员不信任系统，拒绝使用 | 高 | 高 | 从"设备健康监测"叙事起步，以"帮调度员减负"为卖点，不做考核工具 |
| 数据质量不足（GPS采样间隔过长） | 中 | 中 | 第0阶段做数据质量验证，确认后再推进 |
| 主机厂数据平台接口不稳定 | 低 | 高 | 预留本地缓存机制 |
| 调度员用系统但不信系统 | 中 | 高 | 调度员激励机制设计（见上节），撤退线明确 |
| 责任边界不清导致无人敢用 | 中 | 高 | 法务/管理层确认责任边界，作为上线必要条件 |

### 关键依赖

- **总经理支持**：需要他明确向调度团队传达"这个系统是帮你们减负的，不是考核你们的"
- **调度员配合**：需要至少1个愿意试用的调度员作为种子用户
- **数据平台稳定性**：确认主机厂数据接口的SLA
- **产量数据实时化**：建议在MVP期间完成称重传感器安装（否则反馈闭环T+1天，算法迭代太慢）
- **法务/管理层确认**：调度决策责任边界定义，需在系统上线前完成
